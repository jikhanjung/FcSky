# 20260617_053 — SaaS Phase A: 테넌트 모델 + 데이터 backfill

> 계획: [20260617_P03_SaaS_멀티테넌트_전환계획.md](20260617_P03_SaaS_멀티테넌트_전환계획.md)
> 범위: 데이터 모델 도입만. 라우팅·쿼리 스코핑·권한은 Phase B 이후. **동작 변화 없음.**

## 한 일

### 신규 `clubs` 앱
- **`Club`**(tenant): `name`, `slug`(전역 unique·URL 식별자), `logo`, `is_active`.
- **`ClubMembership`**: `(user, club, role=OWNER|STAFF)`, `unique_together(user, club)`.
- admin 등록(Club + 멤버십 인라인).

### 클럽 소유 모델에 `club` FK 추가 (nullable, Phase A)
`Team`·`Player`·`Opponent`·`Match`·`OpponentMatch`·`Notice`·`GalleryItem`·`Award` — 8개.
- `Team.slug` 전역 unique → `unique_together(club, slug)`.
- `Opponent.name` 전역 unique → `unique_together(club, name)`.

### 공유(global) 유지 — club FK 없음
- **`Competition`·`Division`** 은 여러 클럽이 함께 참가하는 실제 대회라 특정 클럽 소유 아님.
- 클럽의 대회 참가는 **기존 `CompetitionEntry`(team→competition→division)** 로 기록
  (team→club 으로 참가 클럽 도출). 새 참가 모델 불필요. ← 설계 결정(사용자 지적 반영).
- `Opponent` 는 **클럽별 분리**(공유 아님)로 결정.

### 데이터 마이그레이션 (`clubs/0002_backfill_fcsky`)
- `Club(slug="fcsky", name="FC Sky")` 생성.
- 8개 모델 기존 행 전부 `club=fcsky` backfill.
- 기존 운영진(is_staff)·관리자(is_superuser) → `ClubMembership(fcsky)`(superuser=OWNER).

## 검증

- `migrate` 정상(밀려있던 pull 분 포함 전부 적용), `check` 무이슈.
- backfill: 전 모델 `club__isnull` **0건**, `Club=fcsky`, `admin=OWNER` 확인.
- 공개 페이지 스모크(`/`, `/matches/`, `/matches/<id>/`, `/stats/`, `/competitions/`,
  `/notices/`, `/gallery/`) 전부 **200** — 단일 클럽 동작 그대로.

## 메모 / 다음(Phase B)

- club FK 는 **nullable 유지** — non-null 확정은 Phase B(쓰기 시 club 자동 주입과 함께).
- Phase B: 미들웨어 `request.club`(슬러그 기반), URL `/<slug>/` 하위화, 전 뷰 쿼리
  club 필터·쓰기 주입. 그 후 non-null 확정.
- 운영 배포는 Phase A 단독으로는 불필요(동작 무변화). 마이그레이션은 다음 배포 시 적용.
