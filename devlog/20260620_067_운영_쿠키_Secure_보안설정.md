# 20260620_067 운영(비 DEBUG) 쿠키 Secure 보안 설정

## 배경
TODO "운영 설정 분리" 점검 중, 설정은 이미 **환경변수 방식**(단일 `config/settings.py` +
`DJANGO_*` 주입)으로 충분해 파일 분리(`base/dev/prod`)는 불필요하다고 결론. 다만
운영 보안 측면에서 앱 레벨 보강이 비어 있었다 — 쿠키 Secure 플래그가 꺼져 있었음.

## 변경 (`config/settings.py`)
```python
if not DEBUG:
    SESSION_COOKIE_SECURE = True   # 세션 쿠키 HTTPS 전용
    CSRF_COOKIE_SECURE = True      # CSRF 쿠키 HTTPS 전용
```
- `SECURE_PROXY_SSL_HEADER`(기존)로 nginx 의 `X-Forwarded-Proto=https` 를 인식하므로,
  프록시 뒤에서도 Secure 쿠키가 정상 설정된다.
- 트리거는 `if not DEBUG`: 운영 compose 가 `DJANGO_DEBUG=false` → 적용,
  개발/테스트는 import 시 `DEBUG=true` 기본 → 미적용(영향 없음).

## 역할 분담 (의도적으로 Django 측 비설정)
- **HTTP→HTTPS 301 리다이렉트·HSTS 는 nginx 담당**(devlog 063). Django 의
  `SECURE_SSL_REDIRECT` 를 켜면 중복일 뿐 아니라 `deploy.sh` 의 직결 HTTP 헬스체크
  (`curl http://127.0.0.1:PORT/admin/login/`)가 301 로 바뀌어 실효성이 떨어진다 → 비설정.
- `X-Content-Type-Options(nosniff)`·`X-Frame-Options(DENY)`·`Referrer-Policy` 는
  Django 기본값으로 이미 적용 → 추가 불필요.

## 검증
- 쿠키 Secure 토글: 개발(DEBUG=true) off / 운영(DJANGO_DEBUG=false) on 확인.

  nginx 담당이라 의도적, W009(SECRET_KEY)는 체크용 더미키 아티팩트(실운영은 .env 주입).
